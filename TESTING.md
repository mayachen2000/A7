# Testing & Verification Specifications (TESTING)

This document maps out the specific testing strategy, individual test cases, verification steps, and actual results for the local Ollama GUI chat application.

---

## 1. Environment & Baseline Tests

### Test Case 1.1: Python Virtual Environment Verification
* **Description**: Verify that Python 3 and a local virtual environment are configured.
* **Steps**:
  1. Open terminal in project root.
  2. Verify that `.venv/` directory exists.
  3. Activate virtual environment: `source .venv/bin/activate` on macOS/Linux.
  4. Run `which python` to verify Python path is within the `.venv`.
* **Expected Result**: Python execution maps directly to the local `.venv`.
* **Status**: **COMPLETED (PASS)**

### Test Case 1.2: Dependency Installation Check
* **Description**: Verify that required library modules are successfully installed.
* **Steps**:
  1. Activate `.venv`.
  2. Check packages list using `pip list`.
* **Expected Result**: `dearpygui`, `requests`, and `python-dotenv` are present.
* **Status**: **COMPLETED (PASS)**

### Test Case 1.3: Ollama Service Status
* **Description**: Verify that the Ollama background daemon is running.
* **Steps**:
  1. Ping the local endpoint in a browser or terminal: `curl http://localhost:11434`
* **Expected Result**: Receives message "Ollama is running".
* **Status**: **COMPLETED (PASS)**

### Test Case 1.4: Model Pull Verification
* **Description**: Confirm that the target lightweight model `smollm:135m` is available.
* **Steps**:
  1. Run command `ollama list` in terminal.
* **Expected Result**: `smollm:135m` appears in the list of local models.
* **Status**: **COMPLETED (PASS)**

## 3. UI Launch & Layout Tests

### Test Case 3.1: GUI Window Initialization (DearPyGui Migration)
* **Description**: Verify that the GUI starts up and displays a stable main window.
* **Steps**:
  1. Activate `.venv`.
  2. Execute `python main.py` or `./scripts/run_app.sh`.
* **Expected Result**: The app launches without warnings or exceptions and displays the main window at 780x640 pixels.
* **Status**: **COMPLETED (PASS)**
* **Observations**:
  - Initial CustomTkinter implementation triggered a startup crash (`AttributeError: '_tkinter.tkapp' object has no attribute 'minimum_size'`) and rendered as a blank unresponsive window under macOS.
  - The frontend was migrated to **DearPyGui (DPG)**, bypassing Tkinter entirely. This resolved the local rendering issue, resulting in a successful window initialization.

### Test Case 3.2: Interactive Components Layout
* **Description**: Confirm header status, scrollable chat logs, send buttons, and text fields are positioned correctly in DearPyGui.
* **Steps**:
  1. Launch the application.
  2. Visually verify alignments, status indicators, and dimensions.
* **Expected Result**: Clean UI alignment with proper space allocation, a dark-themed frame, and reactive labels.
* **Status**: **COMPLETED (PASS)**

## 4. End-to-End Functional Flows

### Test Case 4.1: Manual GUI Chat Interaction
* **Description**: Verify basic chat capabilities within the GUI.
* **Steps**:
  1. Type a message in the input field.
  2. Click "Send" or press "Enter".
  3. Verify that response from the local model appears in the chat display.
* **Expected Result**: Input is appended to the chat display, the input field is cleared, and active replies generated by the local `smollm:135m` model are populated in the logs.
* **Status**: **COMPLETED (PASS)**

### Test Case 4.2: Input Text Validation
* **Description**: Ensure empty inputs do not prompt blank API requests.
* **Steps**:
  1. Leave the text field blank.
  2. Click "Send" or press "Enter".
* **Expected Result**: No outbound request is fired. The status bar displays `"Status: Cannot send an empty message."` to alert the user.
* **Status**: **COMPLETED (PASS)**

### Test Case 4.3: Clear Chat Behavior
* **Description**: Verify the chat window clearing mechanism is integrated.
* **Expected Behavior**: Clicking the "Clear Chat" action button clears the visual conversation log and resets the underlying in-memory message context array (`self.messages = []`).
* **Status**: **IMPLEMENTED & INTEGRATED** (The clear function is fully implemented and mapped to the Clear Chat button, resetting visual logs to `"System:\nChat history cleared.\n\n"` and resetting the client's internal messages array.)

### Test Case 4.4: Ollama Connection Error Handling
* **Description**: Ensure the app fails gracefully if the local background daemon is stopped or unreachable.
* **Steps**:
  1. Stop the Ollama service.
  2. Send a prompt from the GUI.
* **Expected Result**: No crash; displays a clear `System Error` detailing connection issues inside the chat log, and sets the status label to `"Status: Connection/API Error"`.
* **Status**: **IMPLEMENTED / NOT MANUALLY TESTED**
